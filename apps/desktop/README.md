# Kansoku 桌面版

个人交易日志的图表看板，打包成 macOS 桌面 app（`Kansoku.app`），本地跑内核和
图表 UI，不需要开着终端敲命令。

## 下载与安装

去 [Releases 页面](https://github.com/Innei/kansoku/releases) 下载最新一个
`desktop-v*` 版本的 `Kansoku-x.y.z-arm64.dmg`，双击挂载，把 `Kansoku.app`
拖进「应用程序」文件夹。目前只发 Apple Silicon（arm64）版本。

## 首次打开：绕过 Gatekeeper 的"应用已损坏"提示

这个 app 没有走 Apple 付费开发者账号签名（只做了本地 ad-hoc 签名），第一次打开
macOS 会拦一下，提示"无法打开，因为无法验证开发者"或"应用已损坏，无法打开"。这不
是真的损坏，按下面步骤操作一次就好，以后正常双击打开：

1. 在「应用程序」文件夹里找到 `Kansoku.app`，**不要双击**，改成右键（或
   `control` + 单击）。
2. 菜单里选「打开」。
3. 会弹出一个新的确认框（这次带了「打开」按钮），点「打开」。
4. 之后这个 app 再双击就能正常打开，不会再拦。

如果这一步之后还是提示"已损坏"，去「系统设置 → 隐私与安全性」页面往下翻，看看
有没有一条"Kansoku 已被阻止使用"，点旁边的「仍要打开」。

## 配置 Longbridge 凭证

图表数据靠 Longbridge 的行情接口拉取。推荐直接用长桥账号登录：

- 首次打开会看到引导页，点「用长桥账号登录」，会弹出浏览器完成授权，授权成功
  后自动进入正常界面，不需要手动填任何字段。
- 之后凭证会加密保存在本机并自动续期；想换账号或清除，去「设置」页的 Longbridge
  凭证卡片操作。

也可以手动填一套接口凭证（app key / app secret / access token），适合不想走
浏览器授权的场景：

- 申请入口：<https://open.longportapp.com>，用你的长桥账号登录后创建应用。
- 拿到三个字段后，在引导页或「设置」页展开「手动填写 API 凭证（高级）」填入并
  保存即可，不需要重启 app。

## 外部 API 开关（进阶）

「设置」页有一个"外部 API"开关，打开后会在本机起一个带随机 token 的本地 HTTP/WS
服务（默认监听 `127.0.0.1`，不对外网暴露），方便用脚本或其他工具读取图表数据、
订阅实时行情。不需要这个功能的话保持关闭即可，默认就是关闭状态。

## 应用菜单

菜单栏（macOS 顶部）大致是：

| 菜单 | 常用项 |
|---|---|
| **Kansoku**（应用名） | 关于、检查更新…、选择数据目录…、从 repo 导入数据…、设置…（⌘,）、退出 |
| **编辑** | 撤销/复制/粘贴等系统能力 |
| **显示** | 刷新、缩放、全屏、开发者工具 |
| **窗口** | 新建/关闭/切换标签页（⌘T / ⌘W / ⌘⇧] / ⌘⇧[）、最小化 |

- **设置…**：打开或聚焦设置标签页（和标题栏齿轮同一条路径）。
- **检查更新…**：打包版会走 Sparkle 或 GitHub 弱检查；开发模式（`pnpm dev:desktop`）只会提示「开发模式不检查更新」。
- **选择数据目录…**：与设置页「数据目录」同一条路径，见下一节。
- **从 repo 导入数据…**：见「从 repo 导入数据」一节。

## 数据目录

打包版默认把图表和本地库写在 macOS 的应用数据目录里，不在 `.app` 包内：

```text
~/Library/Application Support/Kansoku/
  journal/charts/data/     # 图表 JSON + app.db
  journal/charts/annotations/
  stocks/
```

普通用户不需要改这个路径；装好就能用。

### 绑到本机 trade 仓库（个人常用）

如果你同时用仓库里的 skill / `pnpm dev` / 安装版，希望它们读写**同一棵目录树**，可
以把数据目录指到本机 `trade` 仓库的根（含 `journal/` 的那一层，不要选到
`journal/charts/data` 子目录）：

1. 打开 **设置 → 数据目录**，点「选择…」；或菜单栏 **Kansoku → 选择数据目录…**。
2. 选中仓库根（例如 `…/trade`）。空目录也可以，启动时会自动建好 `journal/`、
   `stocks/` 结构。
3. 确认后**需要重启 app** 才会生效（不会热切换，也不会自动把旧目录里的文件迁过
   来）。

之后安装版与开发/Server 态共用同一棵树：图表、库文件、股票笔记都落在仓库里。

想回到系统默认：设置里点「恢复默认」，同样需要重启。

### 和「从 repo 导入」有什么不同

| | 绑定数据目录 | 从 repo 导入 |
|---|---|---|
| 作用 | 之后读写哪一棵树 | 往**当前**数据根拷贝一份图表 JSON |
| 是否重启 | 要 | 否 |
| 是否持续共用 | 是（长期绑在同一目录） | 否（一次性拷贝，之后各写各的） |
| 是否动 `app.db` | 否（整个库随根切换） | 否（不拷贝库） |

个人长期和仓库共用，请用**绑定**，不要反复导入。

### 不要两边同时写同一棵树

不要同时开**打包安装版**和 `pnpm start` / 本地 server，还都指向同一个数据目录。
两边会抢同一个 `app.db`，可能把库弄坏。同一时间只让一个进程占着这棵树。

### 进阶：环境变量覆盖

启动前设置环境变量 `TRADE_PROJECT_ROOT` 指向某个绝对路径，优先级最高，会覆盖设
置里的自定义路径。适合临时调试；平时用设置页绑定即可。

### 偏好存在哪里

自定义路径记在应用自己的配置文件里：

```text
~/Library/Application Support/Kansoku/data-root.json
```

这份偏好**不会**写进你绑定的仓库，换数据根也不会跟着搬家。长桥凭证、更新状态等
同样始终在系统 userData 下，不进仓库树。

## 从 repo 导入数据

如果你之前一直是跑开发仓库（`git clone` 下来的 `trade` 仓库）攒的图表数据，想
搬到桌面版里接着用，可以用菜单里的「从 repo 导入数据…」（应用菜单
`Kansoku` 下面）：

1. 点这一项，会弹出一个目录选择框，选中你本地 `trade` 仓库的根目录。
2. app 会检查这个目录像不像一个 trade 仓库（有没有 `journal/charts/data` 且里面
   有图表 JSON 文件），不像的话会提示，不会乱复制。
3. 如果桌面版数据目录里已经有同名文件，会先问你是跳过还是覆盖，不会静默覆盖。
4. 复制完会弹一个提示，报告复制了几个、跳过了几个；如果中途有文件复制失败（比如
   磁盘满了、权限不够），也会在提示里列出具体是哪个文件、失败原因，不会假装
   成功。

这只导入**图表 JSON 文件本身**（`journal/charts/data/*.json`），不包含 app 内部
的 sqlite 数据库（评论、AI 用量统计等存在里面），因为这个数据库文件在 app 运行
时是打开状态，直接覆盖有损坏风险，所以现在还不支持——这部分数据桌面版和 repo
各记各的。

这是**一次性搬家**，不是持续同步——导入完之后两边数据就各自独立了，之后在 repo
里新写的图表不会自动出现在桌面版里，需要再导入一次。开发模式下（用
`pnpm dev` 跑起来的那个）本身数据目录就是仓库本体，这一项点了只会提示"无需导入"。
若你希望安装版与仓库**长期共用同一棵树**（而不是偶尔拷一次），请改用上面的
「数据目录」绑定，不要反复导入。

## 更新机制

app 内置 Sparkle 自动更新：启动时和之后定期会静默检查有没有新版本，有的话会提示
下载安装。如果 Sparkle 检查因为某些原因没触发（比如网络问题），还有一层"弱更新
兜底"：app 会额外去 GitHub Releases API 查一次最新版本号，发现落后了会给一个不
强制、可以先忽略的提示条。两层加起来，正常使用下不需要手动去下载新安装包。

## FAQ

**打开是白屏，什么都没有？**
还没配 Longbridge 凭证不会导致白屏——首次打开会看到凭证引导页，不是空白页。真
遇到白屏，大概率是安装包没装完整或者进程没退干净，先完全退出 app（`Cmd+Q`，不
是关窗口）重开一次；如果还是白屏，试试重新下载 dmg 装一遍，反馈时带上具体版本号
方便排查。

**怎么更新到新版本？**
不用手动做什么，参考上面"更新机制"一节，正常联网使用会自动提示。实在等不及也
可以去 Releases 页面手动下载新 dmg 重新装一遍，装新版本会覆盖旧版本，数据不会
丢（数据存在系统的 app 数据目录下，不在 `.app` 包里）。

**AI 相关设置里有的 provider 显示"OAuth"登录，桌面版能用吗？**
可以，但目前桌面版对走 OAuth 登录方式的 provider（例如某些走网页授权的模型
服务）支持还比较有限，稳定可用的是 API Key 方式的 provider。如果某个 provider
在设置页一直显示未配置，优先确认它是不是 OAuth 类型，换一个支持 API Key 的
provider 通常能绕开。
