# trade-gate 交易决策关卡 — 设计文档

日期：2026-07-14
状态：已确认（对话中逐项批准）
灵感来源：@rollingSirius《把美股投资的体力活交给 AI：我的 Claude + IBKR 工作台》第六节"交易决策系统"，按本仓库已有规则本地化。

## 目标

把交易纪律从"自觉"变成"流程"：任何买入/卖出动作发生前，先过一遍写死的检查关卡，打分给出判定；判定与实际执行不一致的记为违规；复盘时能统计"违规的那些亏了多少"。

不做的事：

- 不拦截下单（本仓库不下单，长桥只读）。关卡的约束力来自违规账单，不来自技术拦截。
- 不发明新的卖出规则。卖出端只把用户已有的规则（6/27 持有计划触发线、周期见顶清单、lessons.md）编成可执行的检查项。
- 阶段一不做 web 界面。决策数据先以 JSON 落盘，web 面板是阶段二的只读增量。

## 形态

`.claude/skills/trade-gate/` — 一个 SKILL.md 工作流，模式与 `intraday-signal` 一致：打分规则写死在 SKILL.md 里，由 Claude 按规则执行；数据全部来自 longbridge CLI 与现有 skills；不新写服务、不新建常驻进程。

三个入口：

1. **买入/加仓**：用户说"我想买 X / 加仓 X" → 走买入漏斗。
2. **卖出/减仓**：用户说"我想卖 X / 减 X" → 走卖出触发器。
3. **巡检**：`/trade-gate 巡检`（或"跑一遍卖出检查"）→ 拉长桥全部持仓，对每个持仓跑卖出触发器，输出触发情况汇总。定期跑或暴跌日主动跑。

运行前必读 `journal/lessons.md`（与 `intraday-signal` 同一条纪律）。

## 买入漏斗（六层）

| 层             | 性质   | 内容                                                                                                                                    | 判法                                                                                      |
| -------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| ① 能力圈       | 硬门   | `stocks/{SYMBOL}.md` 六镜头笔记是否存在且非空                                                                                           | 不存在 → 先跑 `stock-deep-dive` 建笔记，本次不打分，直接终止                              |
| ② 可证伪逻辑   | 0–2 分 | 用户一句话写下"为什么会涨"+"什么情况算我错了"                                                                                           | 有逻辑有证伪条件=2；只有逻辑没有证伪条件=0；逻辑含糊=1                                    |
| ③ 估值与周期   | 0–2 分 | 周期见顶清单（`memory` 中 11 条 AI/存储周期信号）当前触发数；股价相对 52 周区间与均线的位置                                             | 触发 0–1 条且位置不极端=2；触发 2 条=1；触发 ≥3 条=0                                      |
| ④ 技术与买点   | 0–2 分 | 拉日 K + 分时：放量/缩量定性（对齐前几日同时段量能）；是否处于爆仓潮未洗净期（韩国三条件，见卖出节）；lessons.md 中相关技术教训逐条核对 | 量价配合且无未洗净爆仓潮=2；有一项存疑=1；缩量阴跌中接飞刀或爆仓潮未洗净=0                |
| ⑤ 仓位与集中度 | 0–2 分 | 拉长桥实时持仓与现金，算本笔加仓后 AI/半导体主线的合计真实敞口（持仓间高度重叠，按穿透后的合计算，不按单票算）                          | 不加杠杆且加仓后现金未清零=2；现金基本用尽（<本笔金额的 50%）=1；需要杠杆或超出名义上限=0 |
| ⑥ 退出预设     | 硬门   | 止损位（具体价格）+ 机械减仓规则，下单前写死                                                                                            | 写不出任一项 → 直接否决                                                                   |

**判定规则**：

- 硬门（①⑥）任一不过 → **否决**，不再看软分。
- 软分（②③④⑤ 合计 0–8）：**≥6 可买（分批）**；**4–5 观望或试探仓**；**<4 否决**。
- 判定输出必须包含：分项得分、判定、依据的关键数据（带来源）、若"可买"则附建议的分批节奏与名义上限。

## 卖出触发器

对单票或全持仓逐项检查，任一触发即在输出中高亮并给出对应动作建议：

1. **持有计划触发线 A/B/C/D**（来源：6/27 持有计划，A 基本面恶化 / B SMH < $560 / C 连续 3 日机构派发 / D 美联储转向加息）。数据来源：longbridge 财报与行情、capital-flow、fred。
2. **周期见顶清单触发数**（11 条，来源：memory `project-ai-memory-cycle-top-signals`）。触发数变化时按用户自己定的减仓规则给建议。
3. **原文六分类兜底**：逻辑证伪 / 估值泡沫 / 机械减仓到点 / 止损触及 / 机会成本 / 仓位失衡。前四条能从数据直接判；后两条列为问句让用户自答。
4. **反向保护——爆仓潮检查**：若当前处于清算未洗净期，无论上面触发几条，都附加提示"不要卖在爆仓潮里，要卖在洗净后的反弹里"。未洗净的判据（lessons 7-13，三取二为洗净）：① SK Hynix / KOSPI 连续两天不创新低 ② 韩国追保比例掉头回落 ③ 美股存储链放量阳线。韩国数据用 `korea-market` skill。

巡检模式输出一张汇总表：每个持仓 × 每组触发器的状态（未触发 / 触发 / 数据未获取到），触发项展开说明。

## 落盘与违规账单

### 决策记录

每次买入/卖出决策（含被否决的）写一个 JSON：

```
journal/decisions/YYYY-MM-DD-SYMBOL-buy.json
journal/decisions/YYYY-MM-DD-SYMBOL-sell.json
journal/decisions/YYYY-MM-DD-patrol.json      （巡检汇总）
```

同日同票第二次决策，文件名追加序号（`-2`）。`journal/` 已被 git 忽略，决策记录随之不入库，与现有日志一致。

字段（买入示例）：

```json
{
  "action": "buy",
  "date": "2026-07-14",
  "emotion": "用户自述一句话，可空",
  "executed": null,
  "falsifier": "用户写下的证伪条件",
  "key_data": [{ "fact": "…", "source": "longbridge kline", "at": "2026-07-14T09:30-04:00" }],
  "plan": { "stop": 58.5, "trim_rule": "…", "size_cap_usd": 600, "staging": "…" },
  "scores": {
    "circle": "pass",
    "logic": 2,
    "cycle": 1,
    "technical": 2,
    "position": 1,
    "exit": "pass"
  },
  "soft_total": 6,
  "symbol": "MU",
  "thesis": "用户原话",
  "verdict": "buy_staged",
  "violation": null
}
```

### 执行核对（不靠自觉汇报）

`executed` 与 `violation` 不在决策当下填写。下次任何入口运行时，自动拉长桥**成交记录**，与未核对的决策记录比对回填：

- 判定"可买"且成交价/数量在预设内 → `executed: true, violation: false`
- 判定"否决/观望"但实际成交了 → `violation: true`
- 判定"可买"但成交偏离预设（超出名义上限、无视分批） → `executed: true, violation: true`，附偏离说明
- 无成交且已过 5 个交易日 → `executed: false`（放弃执行不算违规）

### 违规统计

"算一下违规账单"→ 读全部决策 JSON，对 `violation: true` 的笔目调 `longbridge-profit-analysis` 取已实现/未实现盈亏，输出：违规笔数、违规合计盈亏、守规合计盈亏、对比结论。阶段一由 Claude 现算，不写脚本；若将来 JSON 多到现算吃力，再补一个 stdlib-only 统计脚本。

## 阶段二（本设计只定接口，不实现）

web app（cockpit）加一个只读面板：读 `journal/decisions/*.json`，展示决策时间线、得分分布、违规账单曲线。JSON schema 即上文字段，阶段一冻结后阶段二不改字段、只允许增字段。

## 错误处理

- 长桥调用失败 → 对应层记"数据未获取到"，该层不给分也不猜，判定降级为"观望"并注明缺口；禁止用记忆填数（与 CLAUDE.md 反幻觉规则一致）。
- 韩国数据取不到 → 爆仓潮检查输出"未获取到"，不阻塞主流程。
- 用户拒绝写证伪条件或止损位 → 按硬门规则处理（②记 0 分 / ⑥直接否决），照常落盘。

## 验收标准

1. "我想加仓 MU" → 产出六层打分 + 判定 + JSON 落盘，全程无需用户提供任何持仓/行情数据。
2. "我想卖 DRAM" → 输出 A–D、周期清单、六分类、爆仓潮四组检查结果 + JSON 落盘。
3. `/trade-gate 巡检` → 对长桥全部持仓输出触发器汇总表 + patrol JSON。
4. 决策后实际成交（或未成交）能在下次运行时被自动核对回填，违规判定符合上文四条规则。
5. "算一下违规账单" → 输出违规/守规笔数与盈亏对比。
