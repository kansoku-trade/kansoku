# Settings 页面 UI 重构设计

日期：2026-07-10  
状态：已确认，待实施计划  
范围：apps/web/src/pages/settings/ 与对应页面样式

## 背景

当前 Settings 页面已经具备完整功能：

- 管理 Provider API key 与本机 Codex 登录态；
- 设置主模型；
- 为盘中快评、升级分析、深度研究和追问选择“跟随主模型 / 自定义 / 停用”；
- 测试模型；
- 显示即时保存状态与当日花费。

问题不在功能缺失，而在信息组织。Provider、主模型、用途分配、测试、保存结果和花费沿纵向堆叠，用户需要反复移动视线，才能回答以下四个日常问题：

1. 当前配置是否完整；
2. 每个用途实际使用哪个模型；
3. Provider 凭据是否已经配置；
4. 今天调用了多少次、花了多少钱。

本设计采用“状态总览 + 双工作区”结构，让四类信息同时出现在首屏，并保持现有终端风、高密度和即时保存方式。

## 目标

| 目标 | 成功标准 |
| --- | --- |
| 首屏可判断配置状态 | 不滚动即可看到配置完整度、主模型、Provider 配置数量和今日用量 |
| 缩短常用操作路径 | 更换主模型、设置用途例外、更新 key 和测试模型都能在当前页面完成 |
| 明确状态语义 | 区分“已保存”“配置完整”“本次测试成功”，不把它们统称为“正常” |
| 减少重复控件 | 跟随和停用状态保持单行；只有自定义或编辑凭据时展开输入控件 |
| 保持运行契约不变 | 不改数据库、不改现有 API 响应、不新增轮询 |

## 非目标

- 不增加 Provider 常驻联网探测；
- 不保存“上次测试成功”作为长期健康状态；
- 不增加独立用量页面或用量趋势图；
- 不改变模型分配、凭据加密、即时保存和配置生效语义；
- 不引入新的 UI 库、样式方案或后端接口；
- 不处理 Settings 以外页面的视觉重构。

## 设计原则

1. **事实优先。** 只展示当前数据能够证明的状态。
2. **左主右辅。** 模型分配是高频主工作区，Provider 与异常是辅助工作区。
3. **紧凑默认。** 检查状态不需要打开表单；编辑时才展开控件。
4. **状态集中。** 顶部给出全局摘要，具体问题仍保留在对应行。
5. **错误与主动停用分离。** 红色只表达错误；停用使用弱化样式。
6. **沿用终端风。** 继续使用现有颜色变量、紧凑字号、细边框、低圆角和无阴影结构。

## 页面信息架构

    ┌──────────────────────────────────────────────────────────────┐
    │ 设置  AI 模型、Provider 与用量                               │
    ├──────────────────────────────────────────────────────────────┤
    │ 配置状态 │ 主模型 │ Provider 配置 │ 今日用量                 │
    ├───────────────────────────────────────┬──────────────────────┤
    │ 模型分配                              │ Provider 与凭据      │

    │ ────────────────────────────────────  │ 凭据更新 / 删除      │

    │ 盘中快评                              ├──────────────────────┤
    │ 升级分析                              │ 需处理               │

### 顶部状态条

顶部使用一个连续的状态条，不拆成四张独立统计卡，避免页面变成通用仪表盘。

| 区域 | 内容 | 语义 |
| --- | --- | --- |
| 配置状态 | “配置完整 · 4/4 用途启用”或“2 项需要处理” | 根据本地设置、模型目录和认证配置计算，不代表 Provider 实时可达 |
| 主模型 | 实际模型名与思考档位，或“未设置” | 直接反映主模型草稿的当前值 |
| Provider 配置 | 例如“2 个 key · Codex 已登录” | 分别统计已保存 key 和 Codex 登录态，不使用“3/8 正常”一类可能误导的总分母 |
| 今日用量 | 例如“$1.82 · 127 次” | 沿用 usage-today 的一次性页面加载结果 |

存在异常时，第一格使用琥珀色并显示问题数量；主密钥无法解密时使用红色。完整状态使用绿色，但文案必须写“配置完整”，不得写“系统正常”或“Provider 正常”。

## 双工作区

桌面宽度下使用左宽右窄的两栏布局：

- 左侧约占三分之二，承载主模型与用途分配；
- 右侧约占三分之一，承载 Provider 与“需处理”清单；
- 两栏顶部对齐；
- 窄屏下改为单列，Provider 面板放在模型分配之后。

### 模型分配面板

#### 主模型

主模型位于面板顶部，用较深一档背景和分隔线与用途表区分。保持现有三项选择：

- Provider；
- 模型；
- 思考档位。

同一行保留“测试模型”“清除”和保存状态。更改 Provider 时仍原子地选择该 Provider 的首个模型并把思考档位设为“关闭思考”。清除仍使用确认弹窗，并明确提示跟随用途将暂停。

#### 用途表

用途表固定为五列：

| 列 | 内容 |
| --- | --- |
| 用途 | 盘中快评、升级分析、深度研究、追问 |
| 分配方式 | 跟随主模型、自定义、停用 |
| 当前生效模型 | 实际解析后的模型名与思考档位 |
| 今日用量 | 金额与调用次数 |
| 展开提示 | 自定义状态下表示编辑区已展开 |

三态控件在视觉上沿用分段控件，但实现为语义化 radio group。键盘焦点进入后可用方向键切换，选中项同时使用文字与琥珀色表达。

### 三种用途状态

#### 跟随主模型

- 保持单行；
- 显示实际生效的主模型与思考档位；
- 主模型未设置、模型失效或缺少认证时，生效模型列改为琥珀色问题文案；
- 问题同时进入全局“需处理”清单。

#### 自定义

- 选择“自定义”后，在当前用途行下方展开编辑区；
- 编辑区包含 Provider、模型、思考档位和“测试模型”；
- 继续使用即时保存，不增加全页保存按钮；
- 保存进行中、已保存、保存失败都固定显示在编辑区右下侧，避免布局跳动。

#### 停用

- 保持单行；
- 生效模型列显示“已停用，不会发起调用”；
- 整行弱化，但不使用红色；
- 主动停用不计入“需处理”。

## Provider 与凭据

右侧面板固定列出已保存凭据的 Provider、OpenAI Codex，以及当前主模型或用途正在引用的 Provider。其余未配置 Provider 收进底部“添加 Provider”选择器，避免目录数量较多时把“需处理”区域推出首屏。所有状态仍按认证形态显示事实。

### API key Provider

| 状态 | 文案 | 操作 |
| --- | --- | --- |
| 已保存且可解密 | “已保存” + 脱敏 key + 更新时间 | 更新、删除 |
| 未配置 | “未配置” | 添加 key |
| 无法解密 | “需重新填写” | 更新；全局主密钥异常时提供重置入口 |

“已保存”只表示凭据已经存在且可以解密，不表示远端连接成功。

### OpenAI Codex

- configured：显示“已登录”；
- missing：显示“未登录”，提示在终端运行 Codex 登录；
- error：显示“登录态异常”并进入“需处理”清单；
- 不提供 API key 输入框。

### 编辑方式

- 点击“添加 key”或“更新 key”后，在当前 Provider 行内展开密码输入框；
- 未被引用且尚未配置的 Provider 通过底部选择器添加，选择 Provider、填写 key 后一次保存；
- 成功后重新读取 settings 与 catalog，立即刷新可选 Provider 和问题清单；
- 删除继续使用明确操作按钮；删除会导致正在依赖该 Provider 的用途进入“需处理”；
- 本轮不新增 Provider 单独测试按钮。连接验证仍通过具体模型的“测试模型”完成。

## “需处理”清单

右侧独立区域集中展示当前阻止用途正常工作的事实问题。每条包含：

- 受影响对象；
- 原因；
- 可执行入口。

问题按以下优先级排序：

1. 主密钥异常，已存凭据无法解密；
2. 自定义模型或主模型已经不在目录；
3. 已启用用途依赖的 Provider 未配置认证；
4. 存在跟随用途时，主模型未设置；
5. 已启用用途依赖 Codex 时，Codex 登录缺失或异常。

若没有问题，显示“没有需要处理的配置问题”。该文案不得暗示远端 Provider 已通过实时健康检查。

未被任何已启用用途引用的 Provider 即使没有 key 或没有登录，也只在 Provider 面板显示自身状态，不进入“需处理”，也不影响页面级“配置完整”。

## 状态计算层

新增纯函数模块 deriveSettingsViewModel，把当前设置、模型目录、凭据状态和今日用量转换为页面展示模型。组件不得各自重复推导“是否可用”。

    settings + catalog + usage + 当前角色草稿
                        │
                        ▼
              deriveSettingsViewModel
                        │
             ┌──────────┼──────────┐
             ▼          ▼          ▼
        顶部摘要    用途生效状态   问题清单

每个用途的判定规则：

| 模式 | 完整条件 |
| --- | --- |
| 跟随 | 主模型已设置、模型仍在目录、思考档位有效、对应认证已配置 |
| 自定义 | Provider 与模型已设置、模型仍在目录、思考档位有效、对应认证已配置 |
| 停用 | 永远不算异常 |

页面级“配置完整”表示所有已启用用途均满足上述条件。它不表示模型请求一定成功。

## 组件边界

| 组件或模块 | 单一职责 |
| --- | --- |
| SettingsPage | 并行加载数据、持有角色草稿、协调重新读取、处理整页加载失败 |
| deriveSettingsViewModel | 纯计算顶部摘要、生效模型和问题清单 |
| SettingsStatusStrip | 展示四项首屏摘要 |
| ModelAllocationPanel | 编排主模型和用途表 |
| PrimaryModelEditor | 编辑、测试和清除主模型 |
| RoleAssignmentRow | 展示单个用途，并按模式展开自定义编辑区 |
| ProviderCredentialsPanel | 展示 Provider 状态并管理凭据 |
| SettingsIssuesPanel | 展示排序后的可处理问题 |

角色草稿提升到 SettingsPage，使顶部状态条和用途表能够在用户操作后同步更新。每个角色仍持有独立保存队列；保存失败时，队列把确认值回传给页面并恢复草稿。

角色草稿只在页面首次取得 settings 时初始化。凭据更新触发的 settings 与 catalog 重新读取只合并凭据、主密钥和目录状态，不得覆盖正在编辑、等待保存或已经由 PUT 响应确认的角色草稿。若后续需要支持外部窗口同时修改设置，应另行设计版本冲突处理，不在本轮范围内。

## 数据流

### 初次加载

    GET /api/settings/ai
    GET /api/settings/ai/catalog
    GET /api/settings/ai/usage-today
              │ 并行
              ▼
    deriveSettingsViewModel
              │
              ▼
    统一渲染状态条、模型分配、Provider 和问题清单

usage-today 继续只在页面加载时请求一次，不增加轮询。

### 模型设置即时保存

    用户修改
       │
       ├──> 立即更新页面草稿与摘要
       │
       └──> useSaveQueue 合并连续修改
                  │
                  ▼
           PUT /api/settings/ai/roles/:role
                  │
           ┌──────┴──────┐
           ▼             ▼
         确认值       失败并回滚
                           │
                           ▼
                   原位置提示 + 重试

### 凭据更新

    PUT /api/settings/ai/credentials/:provider
                        │
                        ▼
            重新读取 settings + catalog
                        │
                        ▼
           刷新可选 Provider、摘要和问题清单

### 模型测试

POST /api/settings/ai/test 沿用现有请求体和错误分类。结果仅存在于当前编辑器：

- 成功：显示本次耗时；
- 超时、认证失败、限流或其他错误：显示现有后端返回的安全文案；
- 切换模型、Provider 或思考档位后清除旧测试结果。

## 错误处理

| 场景 | 页面行为 |
| --- | --- |
| settings 或 catalog 初次加载失败 | 显示整页错误和“重试”按钮，不渲染可能误导的摘要 |
| usage-today 加载失败 | 核心配置仍可使用；用量区域显示“暂不可用”和局部重试 |
| 角色保存失败 | 恢复上一个服务端确认值，在对应行显示错误和“重试” |
| 凭据保存失败 | 保留输入值，在 Provider 行内显示错误，不关闭编辑区 |
| 凭据删除失败 | 保持原状态，在原位置显示错误 |
| 模型测试失败 | 只影响当前测试结果，不改变“已保存”状态 |
| 主密钥异常 | 顶部红色问题 + 右侧重置入口，沿用现有确认弹窗 |

所有异步状态都预留固定位置，避免按钮、错误文字和表格列在请求期间跳动。

## 响应式与可访问性

- 桌面保持双栏；窄于约 900px 时改为单栏；
- 窄屏不压缩用途表的关键信息，必要时允许面板内横向滚动；
- 三态控件使用 radio group 语义，并提供键盘操作；
- 错误、警告、完成和停用均同时使用文字，不只依赖颜色；
- 沿用全站琥珀色 focus-visible；
- Spinner 和状态更新使用可读的 aria-live 区域，但避免每次即时保存都打断读屏；
- 尊重现有 prefers-reduced-motion 规则。

## 测试设计

遵循行为测试原则，不为静态标签表、固定角色数组或 CSS 结构写快照。

### 纯计算测试

为 deriveSettingsViewModel 覆盖：

- 主模型未设置且存在跟随用途；
- 主模型或自定义模型已不在目录；
- API key Provider 未配置、已配置、无法解密；
- Codex 未登录和登录态异常；
- 主动停用不计为问题；
- 自定义用途与跟随用途解析到正确生效模型；
- 用量按用途显示，顶部使用服务端 total；
- 问题排序符合优先级。

### 保存行为回归

沿用并扩充保存队列测试，覆盖：

- 快速连续修改只保留最后待保存值；
- 每次请求串行执行；
- 成功后确认值更新；
- 失败后恢复上一个确认值；
- 回滚后再次修改能够重试。

### 浏览器验收

1. 首次打开时，一屏看到配置状态、主模型、Provider 配置和今日用量；
2. 更换主模型后，所有跟随用途与顶部摘要立即同步；
3. 把用途切为自定义，编辑区原地展开并可完成模型测试；
4. 把用途切为停用，整行弱化且不进入问题清单；
5. 删除正在使用的 key，对应用途和顶部摘要立即显示需处理；
6. key 更新失败时输入值不丢失；
7. 保存失败时用途恢复到上一次确认值，并能重试；
8. usage 请求失败时仍能完成所有设置操作；
9. 窄屏下 Provider 面板正确落到模型分配下方；
10. 仅使用键盘可以完成三态切换、展开编辑、测试和凭据更新。

## 验收边界

实施完成必须满足：

- 首屏四项摘要与两栏结构符合本设计；
- 页面不存在未经实时探测却显示“正常”的 Provider 文案；
- 跟随、停用状态默认不渲染下拉框；
- 自定义状态原地展开；
- 全局摘要、用途行和问题清单使用同一纯计算结果；
- 现有 API、数据库和服务端运行行为不变；
- 相关行为测试通过；
- 不覆盖或混入实施开始前已经存在的用户修改。
